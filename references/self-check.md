# 写完后必须过一遍的自检清单

## 一、内容深度

- [ ] 读者是新手：每个新名词第一次出现时是否都解释了？
- [ ] "是什么"模块是否一句话能让人看懂？
- [ ] "原理"模块是否真的讲到了内部机制，而不是又重复一遍 API 列表？
- [ ] "为什么这样设计"是否给出了至少一个对比方案？
- [ ] "怎么用"是否至少一个端到端示例（含输入、代码、输出）？
- [ ] 是否覆盖了基础 + 进阶两类使用场景？
- [ ] 是否列出了至少 1-2 个真实坑点 / 最佳实践？
- [ ] 文中所有"因为 A，所以 B"的句子里，A 是否都已经解释过？（不要把读者还不知道的事实当成不证自明的前提）

## 一·甲 深度下限（达不到必须回去补）

- [ ] 每个核心机制是否都对应到了**具体的源码位置**（仓库 / 路径 / 行号）？
- [ ] 引用的源码路径 / 行号 / 函数签名是否在写之前**实地抓取**核对过，而不是凭记忆？
- [ ] 关键实现是否给了**调用链 / 数据流**（入口 → 函数 A → 函数 B → 数据结构 X），每跳都有路径与行号？
- [ ] 是否至少有 1 张架构图或流程图（Mermaid 等），并在图下解释每个节点对应源码哪里？
- [ ] 是否给了**端到端可复现**的实操示例（环境 / 启动命令 / 输入 / 输出 / 至少一个调参或排错场景）？
- [ ] 命令、配置项、参数默认值是否与官方文档逐项对齐过？
- [ ] 实现 / 源码 / 操作类内容占正文是否 ≥ 50%（不是被纯概念占掉）？
- [ ] 三问能不能答："它内部怎么实现的？现在去仓库找得到吗？我能照着跑起来吗？" 任一答不上来 → 回去补。

## 二、资料源合规性

- [ ] 所有引用链接是否都来自**官方域名**（官方文档站、GitHub 官方组织、RFC、ECMA 等标准组织）？
- [ ] 是否有编造的 URL？（每个链接抽查可访问性 / 拼写）
- [ ] 同一信息至少**两个官方源**交叉印证过？
- [ ] 不一致的地方是否标注了"版本差异"？
- [ ] 不确定的内容是否标了 ⚠️ 待核实，而不是装作确定？

## 三、代码片段

- [ ] 每段代码是否标注了来源（官方文件路径 + 行号 / 文档链接）？
- [ ] 自行编写的代码是否在顶部加了"非官方代码"标注？
- [ ] 自写代码是否与官方 API 签名 / 规范对齐（参数名、返回值、类型）？
- [ ] 每段代码是否带了语言标识？（` ```js `、` ```python ` 等）
- [ ] 关键行是否有中文注释解释"做什么 / 为什么"？
- [ ] 容易踩坑的行是否用 `// ⚠️` 标记？
- [ ] 是否避免了"i++ // i 加一"这类废话注释？
- [ ] 是否避免了"同一段代码贴两遍"——先无注释原版、再加注释副本？（应在第一次给出时就把中文注释写进代码里）

## 四、图片

- [ ] 文中所有图片 URL 是否都来自**官方域名**？非官方图片必须删除。
- [ ] 每张图片下方是否注明了"来源：xxx 官方文档"？

## 五、表格

- [ ] 每张表的**每一行列数**是否与表头一致？
- [ ] 数值类信息是否与官方源对得上？对不上的标 ⚠️。
- [ ] 表格是否真的比文字段落更清晰？只有 2 行的"表"考虑改成列表。

## 六、结构与可读性

- [ ] 是否只有 1 个 `#` 一级标题？
- [ ] `##` 模块编号是否连续（一、二、三…）？
- [ ] `###` 子节编号是否符合所属模块（1.1 / 1.2 / 2.1…）？
- [ ] 每个 `##` 模块顶部是否有"本节回答了什么问题"的引导句？
- [ ] 是否避免了纯概念堆砌（所有概念都对应到了"怎么做"）？
- [ ] 段落是否过长？超过 8 行的段落考虑拆分或改成列表。

## 六·甲 面试视角（必查，缺一不可）

- [ ] 是否包含独立的 `## 九、面试高频考点与陷阱` 模块？
- [ ] 题目总数是否 ≥ 12 道？
- [ ] 是否覆盖全部 6 类题型：基础原理 / 陷阱易错 / 大局系统设计 / 对比取舍 / 源码实现 / 演进趋势？每类是否 ≥ 2 道？
- [ ] 是否至少有 2 道"规模放大后的问题"题（例：N 个 MCP server / 千组件 re-render / 百万 QPS 持久化）？
- [ ] 每道题是否严格使用四段式：**结论先行 / 展开论证 / 延伸追问 / 踩坑提醒**？任何一段缺失即不合格。
- [ ] 每道题是否标注了正文复习锚点（`复习：见 §x.y`）？
- [ ] 答案是否给出**可量化或可落地**的方案（具体参数、数字、策略名），而不是"做缓存"、"加索引"这类空话？
- [ ] 单题答案字数是否 ≥ 80 字？
- [ ] 答案中的源码引用 / 参数 / 数字是否实地核对过？

## 六·乙 篇幅与密度

- [ ] 正文行数是否 ≥ 1500（极简主题 ≥ 800 且已声明）？
- [ ] 正文字数是否 ≥ 25,000（极简主题 ≥ 12,000 且已声明）？
- [ ] `##` 一级模块是否 ≥ 9 个？
- [ ] `###` 子节是否合计 ≥ 35 个？
- [ ] 源码引用片段是否 ≥ 6 段，每段是否都配了独立讲解？
- [ ] Mermaid / 流程图是否 ≥ 2 张？
- [ ] 每个 `###` 子节正文是否 ≥ 80 字？
- [ ] 是否避免了"重复表达凑字数 / 贴大段未讲解代码 / bullet 不展开"三类水文？

## 七、文件输出

- [ ] 文件名是否使用 kebab-case（如 `react-useeffect-summary.md`）？
- [ ] 是否输出为单个 `.md` 文件？
- [ ] 默认是否输出到**用户桌面**（除非用户指定其他路径）？
- [ ] 文件末尾是否有"参考资料"模块？

---

任何一条不通过，**回正文修正**，不要在末尾用"勘误"打补丁。
