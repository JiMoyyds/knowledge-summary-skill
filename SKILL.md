# Knowledge Summary Skill

把一个技术知识点（框架、API、协议、源码模块、语言特性等）整理成一篇**单文件、深度、可读**的 markdown 总结。读者画像是**新手**：他们既需要能看懂概念，也需要看到原理与源码，还要能照着示例上手。

---

## 一、工作流程（必须按顺序执行）

### Step 1 — 解析主题与确认资料源

1. 从用户描述中抽取**主题**与**期望深度边界**。例如"总结 React useEffect"和"总结 React Hooks 全家桶"是不同的范围，需要先界定。
2. 检查用户是否提供了**官方链接 / 文档 / 仓库**：
   - 给了 → 优先用 `web_fetch` 抓取这些链接；
   - 没给 → 用搜索能力定位**官方域名**（例：`react.dev`、`docs.python.org`、`pkg.go.dev`、`developer.mozilla.org`、目标项目的 `github.com/<官方组织>` 仓库）。
3. **绝不编造 URL**。如果搜不到官方源，向用户回报并请其提供。
4. **必须实地抓取**（不能仅凭记忆）：
   - 至少抓 1 份官方文档主页 + 1 份相关源码文件（含具体行号）；
   - 关键函数 / 数据结构 / 参数默认值要在源码里**亲眼看到**再写；
   - 抓到的内容要交叉验证（文档说的接口签名 vs 源码里的真实签名），不一致处标"版本差异"并给出版本号。
5. 先在心里（或简短列出）回答："读者要理解这个主题，最少需要看哪些**文件 / 函数 / 配置项**？" 把这些列入大纲的源码清单。

### Step 2 — 输出大纲并等待确认

在写正文前，先输出**完整大纲**（一级 + 二级 + 计划填入的关键示例 / 源码片段清单），让用户过一遍：

> 默认行为：**先出大纲 → 用户确认 → 再写正文**。除非用户明确说"直接出全文"。

大纲格式参考 `references/template.md`。

### Step 3 — 按层级骨架填充正文

参考 `references/template.md`。骨架要点：

- 用 `##` 作为**一级模块**（概述 / 核心概念 / 工作原理 / 源码剖析 / 为什么这样设计 / 实战使用 / 进阶与边界 / 参考资料）；
- 用 `###` 拆分**子节**，子节数量按主题复杂度增减，过长可继续 `####` 下钻；
- 每个 `##` 模块顶部写一句"本节回答了什么问题"作为引导，方便扫读；
- 禁止只罗列概念，每节必须回答 **能做什么 / 怎么做 / 为什么**。

### Step 4 — 自检与修正

写完后，逐项过 `references/self-check.md` 中的清单。任何不通过的条目必须回到正文修正，而不是只在末尾"勘误"。

### Step 5 — 落盘

输出到当前工作目录，文件名 `{主题英文/拼音 kebab-case}-summary.md`。如果用户指定路径，听用户的。

---

## 二、内容深度的硬性要求

每个总结必须能让读者回答以下问题，否则视为不合格：

1. **是什么** — 一句话定义 + 它解决了什么问题；
2. **核心概念** — 文中出现的所有新名词都要单独解释（"是什么 / 干什么 / 与 X 的区别"），不能默认读者懂；
3. **原理** — 内部机制 / 数据流 / 生命周期 / 关键算法；
4. **源码** — 至少一段官方源码摘录，或 skill 自行编写的关键功能代码（见下文规则），并配中文注释；
5. **为什么这样设计** — 设计动机、与其他方案对比、取舍权衡；
6. **怎么用** — 至少一个端到端示例（输入 → 处理 → 输出 / 效果），覆盖基础用法 + 一个进阶用法；
7. **边界** — 已知坑点、性能注意事项、最佳实践。

---

## 二·甲 深度的下限（达不到就是不合格）

"新手能看懂"≠ "讲得浅"。本 skill 的目标是**读完一篇就能理解具体实现并能上手**，不是写概念科普。每篇总结必须满足：

### 1. 必须深入到代码实现层面

- **不能只停留在"它会做 X"，要回答"它在源码的哪个文件、哪个函数、用什么数据结构、怎么做的 X"**；
- 每个核心机制都要有**官方源码片段**作为依据，并指出 `仓库/路径#行号`（写正文前用 `web_fetch` / 搜索去**实地核对行号**，禁止凭印象写）；
- 如果官方代码太长，截取**关键片段**（10–60 行），把无关分支删掉并用 `// ...` 占位，但保留能完整说明机制的核心结构；
- 关键函数要逐行/逐块讲清"在做什么 / 为什么这么写 / 与上下文怎么衔接"，而不是把代码贴出来就完事。

### 2. 必须给出"怎么实操"

- 至少一个**端到端可复现**的最小示例：环境准备 → 启动命令 / 调用代码 → 预期输入 → 预期输出；
- 命令、配置项、参数名、默认值都必须**与官方文档一致**（核对到具体文档段落）；
- 给出至少一个**调参 / 排错**场景：参数怎么调、调了之后行为怎么变、怎么从日志或返回里验证。

### 3. 必须给出"调用链"或"数据流"

- 用文字 + 代码引用串出**从入口到关键实现的调用路径**（例：`HTTP handler → router.route() → policy.select_worker() → tree.prefix_match_with_counts()`），每一跳都给出文件路径与行号；
- 复杂主题至少给一张**Mermaid 流程图 / 架构图**（用 ` ```mermaid ` 代码块），并在图下方用文字解释每个节点对应源码哪个位置。

### 4. 资料源的"使用强度"

- **必须主动 `web_fetch` 抓取**官方文档与官方源码文件，不能仅凭训练记忆作答；
- 关键事实（参数默认值、数据结构字段、函数签名、版本行为）至少**两个官方源**互相印证（文档 + 源码 / 文档 + Release Notes）；
- 若 `web_fetch` 抓不到（被反爬、404、需要登录），必须**告诉用户**这一点，并标注该处 `⚠️ 未能在线核对`，不要假装核对过。

### 5. 深度自检（写完后回看）

- 把全文当成"答题"——能不能回答："它内部到底怎么实现的？我现在去仓库里能找到它吗？我能照着把它跑起来吗？" 三问任一答不上来，就是写浅了，回去补。
- 概念句和实现句的比例：**实现/源码/操作类内容应占正文 ≥ 50%**，纯概念铺陈不超过 50%。

## 二·补 内容扩展规则（按需延展）

在保证主线清晰的前提下，**遇到下列情况可以、且应该主动扩展**，避免"读到一半卡壳"：

### 何时扩展

- 出现一个**新名词 / 新概念**，不解释会让读者看不懂下一段 → 必须就地补一小节解释；
- 当前知识点依赖一个**前置概念**（例：讲 React Fiber 必然要带一句 Reconciliation 是什么）→ 用一段或一个小节带过；
- 存在容易混淆的**近义概念**（例：进程 vs 线程 vs 协程）→ 加一张对比表或一段并列说明；
- 该知识点有**历史演进**（例：var → let/const、Class 组件 → Hooks），讲清"为什么"时需要带；
- 周边有**强相关生态 / 工具链**（例：讲 webpack 顺带提 loader / plugin 概念）→ 简要点到，给链接；
- 官方文档里出现的**专有术语缩写**（例：HMR、AST、SSR）→ 第一次出现时全称 + 一句话解释；
- **作为论据 / 因果前提的事实**（"因为 X，所以 Y"里的 X）如果读者不一定知道，必须就地补一段解释 X 为什么成立。例：写"因为同一个请求会同时经过 prefill 和 decode"时，必须解释**为什么**会同时经过——是 PD 分离架构下请求先到 prefill 算 KV、再到 decode 生成 token 的工作机制决定的。**严禁把"读者还不知道的事实"当作不证自明的前提直接使用。**

### 扩展的形式

- **小段落内嵌**：一两句话能讲清的，直接在原句后用括号或 `> 提示` 块补充；
- **独立子节**：需要展开的，新开一个 `###` 子节，标题前加 `（扩展）` 或 `（前置）` 前缀，例：
  - `### 2.4（前置）什么是 Reconciliation`
  - `### 5.4（扩展）从 Class 组件到 Hooks 的演进`
- **附录式集中**：与主题相关但偏离主线的内容，集中放到 `## 九、附录` 或在"参考资料"前增加 `## 周边知识地图` 模块，列出相关知识点 + 一句话简介 + 官方链接。

### 扩展的边界（避免跑题）

- 扩展内容**不能比主线还长**，单个扩展子节建议不超过主线对应小节的 1/2 篇幅；
- 每个扩展点要明确说明"它和主题的关系"（前置 / 对比 / 演进 / 周边），不要无脑堆砌；
- 与主题关系弱的内容只给**一句话定义 + 官方链接**，让读者按需深入，不要在本文里展开；
- 扩展同样遵守**资料只采官方源**和**代码处理规则**，不允许因为是"扩展"就放松要求。

## 三、代码处理规则（重要）

代码**不需要本地执行**，但必须保证逻辑正确。

> **核心原则：同一段代码只展示一次，中文注释直接写在代码内（行上或行内），严禁先贴一份"无注释原版"再贴一份"加了注释的副本"。**

### 3.1 来自官方的代码

- 直接摘录，**信任其正确性**；
- 在代码块上方标注来源，例如：
  > 来源：[python/sglang/srt/mem_cache/radix_cache.py#L110](https://github.com/sgl-project/sglang/blob/main/python/sglang/srt/mem_cache/radix_cache.py#L110)
- **第一次给出代码时就把中文注释写进去**，不要再额外贴一份带注释的副本。

### 3.2 官方未提供代码（需要 skill 自己写）

- 只覆盖该知识点的**核心功能**，不写完整可运行项目；
- 准确性验证方式：**逻辑核对 + 与官方 API 签名 / 规范文档比对**（不执行）；
- 代码块顶部明确标注：
  ```
  // 非官方代码：基于 {官方文档/规范名} 编写，演示核心机制
  ```
- 同样在第一次给出时就带好中文注释，不重复展示。

### 3.3 通用要求

- 所有代码块必须带语言标识（` ```js `、` ```python ` 等）；
- 注释解释"做什么 / 为什么"，不要废话注释（如 `i++ // i 加一`）；
- 关键行/容易踩坑的行用 `// ⚠️` 标注；
- **禁止**出现"以下是原代码 …… 加上中文注释后 ……"这种二次展示的写法；如果想强调某几行，单独把那几行抽出来讲解即可，不要把整段代码再贴一遍。

## 四、格式与可读性

| 元素 | 规则 |
|------|------|
| 一级标题 | `# 主题` 仅一个，位于文档顶部 |
| 模块标题 | `## 一、xxx`（中文数字编号，便于阅读） |
| 子节标题 | `### 1.1 xxx` |
| 重点标记 | `**加粗**`、`> 提示框`、必要时用 HTML `<span style="color:#c0392b">…</span>` 标红（仅在渲染器支持彩色时生效） |
| 图片 | **只允许引用官方域名**的图片（如 `react.dev/...`、`docs.python.org/...`）。其他一律不插。引用时附原文档链接。 |
| 表格 | 必须做列一致性自检（每行列数相同、表头对齐）；不确定的数值标 ⚠️ |
| 不确定信息 | 标注 `⚠️ 待核实` 而不是猜测 |
| 链接 | 全部用 `[文字](URL)` 形式；末尾"参考资料"集中列出 |

---

## 五、禁止事项

- ❌ 编造 URL、API、参数名、版本号；
- ❌ 插入非官方域名的图片；
- ❌ 只讲概念不讲怎么做；
- ❌ 代码不带注释 / 不标来源；
- ❌ 同一段代码贴两遍（先贴无注释原版，再贴一份加了中文注释的副本）；
- ❌ 默认读者懂某个名词而不解释；
- ❌ 在"因为 A，所以 B"的论述里，把读者尚不熟悉的 A 当作不证自明（必须先解释 A 为什么成立，再得出 B）；
- ❌ 跳过大纲确认环节直接写全文（除非用户明确要求）；
- ❌ 仅凭训练记忆作答而不实际抓取官方文档/源码；
- ❌ 给出的源码路径 / 行号 / 函数签名未实地核对；
- ❌ 只讲"它会做 X"而不指出"在哪个文件、哪个函数、用什么数据结构怎么做的 X"；
- ❌ 没有可复现的端到端实操示例（启动命令 / 调用代码 / 预期输入输出齐备）。
